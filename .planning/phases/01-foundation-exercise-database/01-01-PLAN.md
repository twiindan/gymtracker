    2. Create supabase/migrations/001_create_exercises.sql with the exercises table:
       ```sql
       -- Tracking type enum
       CREATE TYPE tracking_type AS ENUM ('reps', 'duration', 'distance', 'bodyweight');

       -- Exercises table
       CREATE TABLE exercises (
         id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
         name TEXT NOT NULL,
         primary_muscle TEXT NOT NULL,
         secondary_muscles TEXT[] DEFAULT '{}',
         equipment TEXT,
         tracking_type tracking_type NOT NULL DEFAULT 'reps',
         is_custom BOOLEAN NOT NULL DEFAULT false,
         wger_id INTEGER UNIQUE,
         created_at TIMESTAMPTZ DEFAULT now()
       );

       -- Full-text search index on exercise names (trigram for fuzzy matching)
       CREATE EXTENSION IF NOT EXISTS pg_trgm;
       CREATE INDEX idx_exercises_name_trgm ON exercises USING gin (name gin_trgm_ops);

       -- Index for muscle group browsing
       CREATE INDEX idx_exercises_primary_muscle ON exercises(primary_muscle);

       -- Index for filtering custom vs built-in
       CREATE INDEX idx_exercises_is_custom ON exercises(is_custom);
       ```

    3. Create scripts/seed-exercises.ts — a TypeScript script that:
       a. Fetches all muscles from wger API: GET /api/v2/muscle/ — build a map of id -> name_en (fallback to name if name_en empty)
       b. Fetches all equipment from wger API: GET /api/v2/equipment/ — build a map of id -> name
       c. Fetches exercises with pagination: GET /api/v2/exercise/?limit=100&offset=N
          For each exercise batch, also fetch exercise aliases for English names: GET /api/v2/exercisealias/?exercise={id}&limit=1
          Use the first alias as the exercise name (prefer English — language_id 2 if available)
       d. Transform each exercise:

          - equipment: resolve equipment[0] from equipment map, or null
          - tracking_type: infer from equipment — if equipment includes 'none (bodyweight exercise)' then 'bodyweight', else 'reps'
          - is_custom: false (built-in)
          - wger_id: the exercise id from wger
       e. Batch insert into Supabase using .upsert() with onConflict: 'wger_id' for idempotency
       f. Log progress: "Fetched N muscles, M equipment, seeded X exercises"

       - Handle pagination: loop until next is null
       - Rate limiting: add 100ms delay between API calls to be respectful
       - The script should be runnable with: `npx tsx scripts/seed-exercises.ts`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase | Single-user app with anon key — no auth layer. All data is public to anyone with the URL (acceptable for personal app) |
| Seed script → wger API | Unauthenticated public API — no secrets transmitted |
| User input → exercises table (custom exercises) | Custom exercise names must be sanitized to prevent XSS when rendered |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | exercises table (custom exercises) | mitigate | Sanitize custom exercise names with DOMPurify or server-side escaping before insert; use parameterized queries via Supabase client (automatic) |
| T-01-02 | Information Disclosure | Supabase anon key in .env.local | accept | Single-user personal app; anon key is designed for client-side use. RLS policies not needed (no multi-user) |
| T-01-03 | Denial of Service | wger API seed script | mitigate | Add 100ms delay between API calls; handle HTTP errors with retry logic (max 3 retries with exponential backoff) |
| T-01-04 | Spoofing | Exercise name from wger aliases | accept | Public API data — no auth. Trust wger as authoritative source. Validate name is non-empty string before insert |
</threat_model>
