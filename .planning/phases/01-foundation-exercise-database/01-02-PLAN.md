<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Supabase (exercise insert) | Custom exercise names and fields are user-controlled input written directly to DB |
| Browser → Supabase (exercise read) | All exercises readable by anyone with anon key — acceptable for personal app |
| Client-side filtering | Search and filter run in browser — no server-side query injection risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Injection | AddExerciseForm → exercises table | mitigate | Supabase client uses parameterized queries automatically — no SQL injection possible. Add client-side name validation (max 100 chars, alphanumeric + spaces + basic punctuation) |
| T-01-06 | XSS | ExerciseCard rendering exercise.name | mitigate | React auto-escapes text content by default. Never use dangerouslySetInnerHTML. Exercise names rendered as plain text in JSX |
| T-01-07 | Authorization | Built-in exercise modification | mitigate | UI does not render edit/delete buttons for exercises where is_custom=false. Supabase RLS can add server-side protection later if needed (accept for now — personal app) |
| T-01-08 | Integrity | Duplicate custom exercise names | mitigate | Client-side uniqueness check via Supabase query before insert; database-level unique constraint not added (wger_id is unique for built-in, custom exercises can share names — user responsibility) |
</threat_model>
